The one-minute version

What we know, in three sentences

The camera is a Throughtek Kalay (successor to CS2 Network PPPP) device with a custom C-layer called "CBS" on top, that registers with three Aliyun supernodes via UDP/32100 and calls a hapseemate.cn HTTPS API for user-binding lookups. Its only on-wire defense against local interception is a DNS-rebinding filter that rejects private-IP answers for control-plane hostnames, which we bypass by using a non-RFC1918 sink IP and then DNAT-rewriting the traffic at our router. With a TLS MITM in place (the camera does not cert-pin), we now see every byte the camera produces in cleartext — including a brand-new JSON response schema with a typo ("mssage":"success") we would never have guessed.

The device under test

OEM identity and what it tells us

The camera is branded as "Cloudbirds" (by Dayunlinks, the parent company), but internally it’s a Qianniao-OEM device using a Throughtek Kalay SDK. The DID encoding alone reveals most of the stack.

The DID: CFEOA-417739-RTFUU

prefix5 lettersCFEOA
serial6 decimal digits417739
suffix5 lettersRTFUU

This is the canonical CS2/Kalay UID form AAAAA-NNNNNN-BBBBB. The 5-letter prefix CFEOA maps to a specific OEM in Throughtek’s vendor registry — each OEM gets its own 80-character vendor init key (we found three in P2pSDK.java). The key is what makes this cam register with Cloudbirds’ supernodes specifically rather than some other Kalay customer’s.

The serial 417739 fits in 24 bits and gets encoded as a 32-bit big-endian integer in LAN discovery responses: 00 06 5f cb = 0x00065fcb = 417739. The 8 bytes of zero-padding in the encoding suggests the OEM once considered 8-character prefixes/suffixes and reserved the space.

Kalay wire format

The simplest possible protocol header

Every Kalay packet is a 4-byte header then a payload. The header is fixed across all message types; the payload format depends on the type.

0xF1magic byte
(always)
typeu8
message type
len_hiu8
len_lou8
body length (BE)
< body >0 to 65535 bytes · format depends on type

The 0xF1 magic is the lowest-level identifier — any UDP packet starting with F1 and roughly matching (len+4) == packet_size is going to be Kalay. That’s how Wireshark and tshark can identify these without a dedicated dissector.

Length encoding

Bytes 2-3 are an unsigned 16-bit integer in network byte order (big-endian) representing the body size in bytes. Zero-length bodies (like HELLO) are completely valid — the packet is just the 4-byte header.

A packet of type 0x12 (DEV_LGN) with a 44-byte body looks like:

DEV_LGN packet · 48 bytes total
f1 12 00 2c e3 86 b5 b9 df 2e de ce 90 aa 57 fd 34 06 30 21 41 1f 42 22 e0 a3 d4 d8 ea 2c 6d 41 32 84 3e 82 b8 b8 14 34 a9 8e 28 26 6a 5e 3b dc 7 │ │ │ │ │ └─ body length = 0x002c = 44 bytes │ └─ type 0x12 = DEV_LGN_CRC └─ magic 0xF1

Why so simple?

CS2 Network wrote this protocol in C in the early 2000s for embedded devices with tiny flash and tiny RAM. Every byte counted. So:

  • No TCP — UDP only, to avoid state overhead
  • No TLS — most bodies get a hand-rolled XOR obfuscation with a static vendor key
  • No variable-length type field — one byte is enough for 256 types
  • No length for the header itself — it’s always exactly 4 bytes
  • Bodies for high-volume types (HELLO, ALIVE) are empty

Twenty years later, the same protocol ships in millions of cameras because it works and nothing has replaced it. The "Kalay" rebrand by Throughtek added some new message types (0xF9 ReportSessionRdy is Kalay-era) but the 4-byte header survived unchanged.

The one clever vendor defense

DNS rebinding protection

The one meaningful security feature in this camera is a filter that rejects RFC1918 answers for control-plane hostnames. It’s specifically designed to prevent exactly what we’re doing. Bypassing it is one extra line in the attack chain.

How the filter works

The camera’s firmware wraps getaddrinfo() with a check: after resolving a known control-plane hostname, if the returned A record is in 10/8, 172.16/12, or 192.168/16, the firmware **silently refuses to connect** and retries DNS. From the outside it looks like the camera is resolving DNS correctly but never attempting a connection.

We confirmed the filter exists by comparing two test runs:

  • dnsmasq → 192.168.5.233: zero TCP SYN attempts
  • dnsmasq → 9.9.9.9: immediate SYN to 9.9.9.9:443

The only difference is the IP class. The camera never saw either address — dnsmasq is the one doing the resolution. But the firmware checks the resolved value.

Our bypass

Instead of pointing DNS at our Mac’s LAN IP directly, we point at any non-RFC1918 IP. The camera happily "connects to the internet", and a router rule silently rewrites that packet to our Mac.

dnsmasq.conf · on UDM
# Any non-RFC1918 IP works. 9.9.9.9 is Quad9’s real # public DNS — noisy but it proves the principle. # For production use 203.0.113.x (TEST-NET-3, unroutable). address=/cloudbirds.cn/9.9.9.9 address=/hapseemate.cn/9.9.9.9 address=/hapsee.cn/9.9.9.9
iptables · on UDM
# Rewrite cam’s 9.9.9.9 traffic to our Mac iptables -t nat -A PREROUTING \ -s 192.168.5.37 \ -d 9.9.9.9 \ -j DNAT --to-destination 192.168.5.233 iptables -t nat -A POSTROUTING \ -s 192.168.5.37 \ -d 192.168.5.233 \ -j MASQUERADE
Side note — the "hole" in the defense: the filter only runs for known hostnames. The camera also has a hardcoded supernode IP (123.56.74.245) that it contacts without going through DNS at all. That one is immune to DNS-level tricks but vulnerable to the exact same IP-level DNAT rule — just with the hardcoded IP as the destination. We added both.

1

dnsmasq override

One config file change on the UDM. Points every cam-cloud hostname to a non-RFC1918 sink (9.9.9.9 for now, TEST-NET-3 for cleanliness).

2

iptables DNAT + MASQ

Rewrites the cam’s packets destined for the sink IP to our Mac. Critically, also rewrites return-path source so the cam’s conntrack matches.

3

mitm_cbs_proxy.py

Full TLS MITM on port 8443. Self-signed cert with SANs for every cloudbirds hostname. Forwards decrypted requests to real AWS, logs everything.

4

mitm_supernode_proxy.py

UDP proxy on port 32100. One upstream socket per cam session (round-robin across 3 real supernodes). SIGUSR1 injection for crafted packets.

5

Kernel conntrack

The glue nobody sees. Both MITMs rely on Linux’s automatic NAT reverse mapping so replies go back to the cam with the original "cloud" source IP.

!

Not in the chain: cert pinning

The single biggest vendor defense we didn’t have to fight. The cam accepts any cert for *.hapseemate.cn that validates hostname — ours does.

The CBS control plane

Real cloud API · decoded via MITM

Until the MITM was in place, this traffic was a black box. Now we can see the exact request URL, exact response schema, and exact content-type the vendor uses.

cam → hapseemate.cn · HTTPS request
GET /preadd/didBindUserId? relatInfo=eTgjfHOm%2B3jo…%3D& // b64 of encrypted blob did=Nb72w3ZMX2AtpU74…%3D& // b64 of encrypted blob utcTime=1776203762& // Unix time, NTP-synced devType=1& // 1 = camera class netType=1& // 1 = Wi-Fi eleType=1 HTTP/1.1 // 1 = indoor element Host: user.hapseemate.cn Accept: */*
hapseemate.cn → cam · HTTPS response
HTTP/1.1 200 Date: Tue, 14 Apr 2026 22:20:41 GMT Content-Type: text/plain;charset=ISO-8859-1 Content-Length: 35 Server: elb // ← AWS ELB, not Alibaba Vary: Origin {"mssage":"success","status":"200"} └─ typo, preserved verbatim
The typo is load-bearing. The real backend is Java (based on the Spring-style endpoint path naming and ELB signature). Someone wrote mssage instead of message early on, shipped it, and now every cam firmware in the field parses the misspelled key. Fixing the typo would break every client. A hand-crafted fake response with the correct spelling {"message":"success","status":"200"} would not satisfy the cam — we checked. This is the kind of detail you’ll never learn from static analysis.
Also load-bearing: the Java BaseRepo class in the Android app has 23 fields — including ad-display config (advType, isNative, coldStartInr, etc). None of those fields actually appear in the cam firmware’s parser. The Android app deserializes them because Java is strict; the cam uses a permissive C JSON parser (kcJSON) that just looks for the fields it needs and ignores the rest. Two clients, two schemas, one server — and the "schema" you get in each case depends entirely on what the client is willing to parse.

Summary

Key architectural takeaways

①

Simple protocol, complex ecosystem

Kalay itself is ~20 years of legacy C, with a 4-byte header and ~20 message types. The complexity is all in the cloud topology: three supernode regions, two HTTPS APIs, a hardcoded fallback IP, and NTP wrappers.

②

The defense is DNS-shaped

The vendor put their security budget into a DNS-rebinding filter and skipped cert pinning entirely. One router rule mooted the filter; lack of pinning made the rest trivial. Inverted from how most security products recommend allocating effort.

③

Reality reveals fields static analysis can’t

The "mssage" typo was impossible to learn by reading the APK — it only appeared once we MITM’d real traffic. Similarly, the 0xF9 message type only fires after HELLO_ACK echoes a non-RFC1918 IP. Running the real thing beats reading the code.

④

The MITM pivot was the key

We spent hours guessing response schemas. The moment we pointed the MITM at the real cloud and let Alibaba/AWS compute the responses for us, progress went from "one bit per guess" to "full cleartext stream."

⑤

OEM patterns are universal

The same libPPCS_API.so, libCBSClient.so, and 0xF1 magic byte ship across dozens of brands — HapSee, V360, Yoosee, Sricam, YI, and more. Everything in this explainer is portable.

⑥

Cheap cams are not hopeless

The gap between "black box" and "fully understood" turned out to be ~3 evenings of work with off-the-shelf tools (jadx, tshark, Python, iptables). The hardware is fine. The vendor software is avoidable.
