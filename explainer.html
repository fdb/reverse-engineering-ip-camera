Wire format · discovery probe

The “YH” broadcast — only 4 bytes

The phone blasts this tiny packet four times, 500 ms apart, to 255.255.255.255:18523. Any Qianniao-family camera on the subnet replies.

0x59'Y' magic
0x48'H' magic
0x01version
0x01subtype

Reply layout (280-byte max buffer)

Fields are ASCII, zero-padded and trimmed. The code validates by checking bytes 0–1 still equal 'Y','H', then slices at fixed offsets:

bytes 0–4header & magic
bytes 4–20IP string · 16 B
bytes 20–84version / model (unparsed)
bytes 84–90MAC · 6 B
bytes len−24 … len−4DID · 20 B
SearchDeviceByMulticast.kt · constructor + send
val buf = ByteBuffer.allocate(4) .putShort(18521) // 0x4859 → 'Y','H' (big-endian) .put(0x01) // version .put(0x01) // subtype val pkt = DatagramPacket(buf, buf.size, InetAddress.getByName("255.255.255.255"), 18523) repeat(4) { socket.send(pkt); Thread.sleep(500) }

The bypass

Privacy-preserving setup — no app needed

Everything the camera cares about is local. The cloud round-trip is the app’s choice, not the camera’s requirement. Here’s the minimal sequence that never leaves your LAN.

✓

1 · Join the HAP-xxxxx AP

SSID starts with HAP-, is ≥ 21 chars. No password on the AP itself — the "123" you’ll find in the code is the PPPP admin password, not a Wi-Fi one.

ApDirectHintFragment.kt
◉

2 · Listen locally on 8899

Our probe.py passive does exactly what the app does: bind UDP/8899, wait for the cam’s unsolicited beacon, parse its DID and current IP.

SearchDeviceByUDP.kt · probe.py
✦

3 · Generate the Wi-Fi QR yourself

Payload is literally "S" + ssid + "\n" + pwd + "\n". Any qrcode library produces it. No token, no cloud, no Aliyun.

WifiScanQrcodeFragment.kt:257
⌘

4 · Probe again on the real LAN

Run probe.py active. The cam replies with the new IP. From there, PPPP login at cam-ip:8000 with DID + "123".

SearchDeviceByMulticast.kt · libPPCS_API.so
Net effect: the Chinese cloud never hears about your SSID, password, or device ID. You still get full local video via PPPP. Outbound firewall rules can block anything else the cam tries to phone home.
Summary

Key findings, in five bullets

①

It’s an OEM clone

Dayunlinks / Cloudbirds is a skin. The real stack is Qianniao + CS2 Network PPPP, shared with dozens of other cheap cams. What works here likely works on its cousins.

②

Local protocol is trivial

Discovery is a 4-byte UDP broadcast with a “YH” magic. Reply layout is fixed offsets — ~50 lines of Python reproduces the entire scanner.

③

Provisioning leaks creds

The app POSTs your home SSID + password to the vendor cloud before encoding the QR. Not necessary for pairing — it’s pure telemetry.

④

The bypass is a 30-line script

Listen on 8899, generate the QR with the exact string the app would have used, done. No APK required, no Google Play data-sharing agreement.

⑤

Next unknown: PPPP frames

Discovery and pairing are solved. The remaining RE task is the DevLgn packet structure in libPPCS_API.so so we can pull raw H.264 without the app’s player.

☁

Other phone-home vectors

Even with a clean setup, the cam will try Aliyun IoT (libiot.so) and whatever libCBSClient.so is talking to. Block egress on the LAN side, not just on the phone.
